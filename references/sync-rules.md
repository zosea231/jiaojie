# 交接文档同步规则

触发方式：用户在对话中明确说出类似"同步进度到交接文档"/"sync progress
to the handoff docs"/"更新交接文档"时，立即执行本规则，而不是继续往下做
任务或空谈"已经完成"。

## 每次同步要做的事
1. 回顾本次会话里发生的、下一个agent需要知道的事实（不是回顾整个对话，
   只挑对下一个人有用的结论）。
2. 按内容类型分别路由到对应文件，而不是把所有东西堆进一个文件：
   - 任务目标变化/明确 → `.ai/brief.md`
   - 方案/设计决策、最小修改建议 → `.ai/plan.md`，同时更新状态勾选项
   - 审查发现（分 P0/P1/P2） → `.ai/review.md`，同时更新状态
   - 新增的任务专属验收标准 → 追加 `scripts/checks/<name>.sh`（不改主流程），
     并在 `.ai/decision-log.md` 记一行原因
   - 发现但本次不处理的问题 → `.ai/backlog.md`（追加一行，不要覆盖旧记录）
   - 非显而易见的"为什么这样做" → `.ai/decision-log.md`（追加新条目，
     不要修改或删除已有条目）
3. 只写结论和当前状态，不要写成对话摘要或流水账；假设读者是一个完全
   没看过这次对话、只会读这几份文件的新agent。
4. 同步完成后，用一句话告诉用户改了哪些文件、现在状态是什么，方便
   用户判断该切到哪个agent，例如：
   "已同步：plan.md 标记为'已实现待审查'，backlog.md 新增1条，
   decision-log.md 新增1条决策记录。可以切到 Claude Code 做审查了。"

## 不要做的事
- 不要在没被要求同步时，主动大段改写 `.ai/` 文件——只有明确触发词或
  自然完成一个阶段时才写。
- 不要重写 `decision-log.md`/`backlog.md` 的历史条目，只追加。
- 不要因为"顺手"就去动本轮任务范围之外的文件。
